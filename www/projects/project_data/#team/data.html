#DataSources

How did you collect the data?

  • Counts: We are using the counts endpoint of the Twitter Full Archive Search API to collect the number of tweets per day that contain one of the given hashtags.
  • Events: To create our Events dataset, we selected 20 events for each movement from our research, 10 of which are categorized as “triggering” (shootings, scandals, etc) and the other 10 categorized as “activism” (marches, protests).
  • Movements: We selected three social movements to focus our project on and created a table including data (such as the time range and associated hashtags) about each one.

Are the sources reputable?

  • Events: We recognize that the selection of events included in our events database is biased, as there is not a truly unbiased way to find events for the social movements. We conducted our own research on the web to find significant events for each movement.
  • Counts: Yes, we are using data given directly from Twitter's API which accurately represents Twitter data.
  • Movements: We recognize that the selection of movements is biased, but considering the scope of our project, choosing three movements ensures that our project is feasible.

How did you generate the sample?

  • Our events database is relatively small and contains samplias, as we manually selected events, and our reasoning is explained below.

Are there any other considerations you took into account when collecting your data?

  • Events: We considered using quantitative metrics to select the events for the events database and looked into using the New York Times API to select the events with the highest number of articles within three days after the event. However, in this case, we would still have to compile a list of events from which to choose a sample of events, and this list of events would still be biased due to its manual selection. Thus we decided against using this API and opted to use 20 events per movement, which is a higher number than our original plan to use three events per movement, since analyzing more events may be able to counteract some of the bias resulting from manual selection.
  • Counts: Private or deleted tweets are not included in the count totals, which could potentially skew the data by focusing on public users (and their behavior), rather than all users.

#DataCleaning

How clean is the data? Does this data contain what you need in order to complete the project you proposed to do?

Our Events data is organized in a .csv file, which contains all the data we need pertaining events. We expect the output from the counts endpoint from the Twitter Full Archive Search API to be sufficiently clean for our purposes, as the output field will contain the counts, which we can just extract.

How many data points are there total? How many are there in each group you care about (e.g. if you are dividing your data into positive/negative examples, are they split evenly)? Do you think this is enough data to do what you hope to do?

There will be 60 data points total, as we have 20 events per movement, and we will be analyzing a total of three movements. Of the 20 events for each movement, 10 are classified as triggering and 10 are classified as activism events (e.g. protests, marches, campaigns). This is a relatively small set of data points, but we plan to use t-tests to compute our statistics, so this size of data should be sufficient.

Are there missing values? Do these occur in fields that are important for your project's goals?

We are missing the count values for the tweets per event. For now, we have collected temporary counts per day using the Full Archive Search API, which returns a maximum of 100 tweets.

Are there any data type issues (e.g. words in fields that were supposed to be numeric)? Where are these coming from? (E.g. a bug in your scraper? User input?) How will you fix them?

We have not run into this issue.

Do you need to throw any data away? What data? Why? Any reason this might affect the analyses you are able to run or the conclusions you are able to draw?

The Full Archive Search API returns all of the relevant data for a given tweet, however, we are throwing away data that is not relevant to our project. This data includes information about the user and information about interactions with the tweet. This should not affect our data analysis because these fields are not relevant.


#Challenges

One major challenge we faced is the limitations of the Twitter API, which we realized (after much research) only gave a limited number of tweets that were posted within the last seven days. Per our project goals, we want all the tweets within specific time spans of movements and have a count for all these tweets, so not having access hinders our data collection tremendously. We are currently in the process of working with Ellie to get access to the Twitter Premium API. Once we have this access, we will be able to retrieve the daily counts of tweets for a period of several years using a script that we have already written. Also, this will help us overcome the challenge of only having a limited number of API calls since the counts endpoint from the Full Archive Search API enables us to gather the number of relevant tweets with a single call.
